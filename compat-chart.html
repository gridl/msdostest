DOSBox MS-DOS application compat testing chart
MS-DOS testing repository: https://github.com/joncampbell123/msdostest

DOSBox-X refers to the DOSBox-X project
DOSBox-SVN refers to the DOSBox project

Rules regarding PASS or FAIL:
- If a game crashes, faults, or shows behavior that suggests a problem, that is a FAIL.
- Known faulty behavior in the game itself doesn't count as failure. Future development may offer off-by-default hacks or workarounds in the emulator for the problem.
- Known faulty behavior that occurs on real hardware doesn't count as failure, because it means DOSBox is emulating real hardware accurately and the game is at fault.
- Issues that are DOSBox-SVN's fault, that the DOSBox SVN developers have expressed an unwillingness to fix or accept patches to fix, do not count as failure.

Common problems with DOSBox SVN so far:
- Lack of support for "goldplay" Sound Blaster playback. Audio is rendered "muffled" in DOSBox SVN.
- LPT DAC output doesn't work. However code explicitly written for the Disney Sound Source works.
- PC speaker emulation on/off and frequency changes are limited to 1ms precision. This affects some games as well as old Apogee titles like Duke Nukum.

Common problems to both DOSBox-X and DOSBox-SVN:
- Having UMB or EMS enabled can cause crashiness in some games.
- Some games require XMS to be enabled, will call to a NULL pointer otherwise.
- Some games will crash or malfunction if loaded too low (below 64KB) in memory. Try loadfix or the minimum mcb segment option.

Problems that do NOT count as failures against DOSBox SVN, because they are bugs in the game itself:
- Audio halting or interrupt problems with GUS or SB related to a failure to unmask the IRQ (gamecoder laziness, NOT SVN's fault).
- Hangs related to undefined or reserved bits in VGA emulation that the game really ought to ignore.
